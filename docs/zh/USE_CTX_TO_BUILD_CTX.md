# 用 CTX 构建 CTX
????????????????,??????????

## 目标

将 CTX 用作产品自身演进的官方认知仓库。

## 在此过程中 CTX 应保存的内容

- 产品目标
- 结构化 backlog
- 价值与架构假设
- 真实测试证据
- 技术与产品决策
- 每次迭代的结论
- 每次提交的认知快照

## 当前缺失

- UI 中的提交可视化对比
- 时间线更丰富的父/子导航
- run 与工件关系的更好捕获
- 明确的假设评分模型
- 更闭合的日常 CTX 操作流程

## 推荐的操作流程

### 1. 打开循环

```powershell
ctx status
ctx graph summary
ctx log
```

### 2. 创建迭代目标

示例：

- 加固 viewer
- 改进认知 diff
- 定义假设评分
- 准备 V1 试点

### 3. 拆解为任务与假设

每次迭代应包含：

- 1 个清晰目标
- 2 到 5 个任务
- 1 到 3 个关键假设

### 4. 工作过程中记录证据

示例：

- 测试结果
- 发现的限制
- viewer 反馈
- benchmark
- 使用摩擦

### 5. 用决策与结论闭环

不要留下未回答的问题：

- 我们学到了什么
- 我们改了什么
- 为什么要改
- 还有什么未关闭

### 6. 认知提交

每个有价值的工作块都应以以下结束：

```powershell
ctx commit -m "<block result>"
```

### 7. 自主延续

当操作者或模型仅收到 `continua` 时，应：

1. 重新读取 `ctx status`、`ctx graph summary` 与 `ctx log`
2. 选择当前主导目标
3. 选择最阻塞或价值最高的任务
4. 产出真实证据
5. 记录结论
6. 关闭认知提交

如果 CTX 仓库已明确下一步，不要等待手动指令。

## 建议的认知工作结构

### Goals

- 可用的本地 V1
- 可操作的 viewer
- 认知捕获方法
- 验证试点

### Tasks

- 改进 UI
- 打包发布
- 安装 CLI
- 测试 demos
- 定义评分

### Hypotheses

- 可视化追溯减少返工
- viewer 加速历史理解
- 简单评分提升决策质量
