# Internal Structure of `.ctx/`
If a language model and its agent lose context, this is the tool you need.

This document describes CTX local storage structure.

The `.ctx/` folder is the persisted cognitive repository on disk. It contains configuration, active state, reproducible history, and operational artifacts generated by the system.

## Objective

The `.ctx/` structure is designed to:

- persist structured reasoning, not raw conversations
- enable reproducible cognitive commits
- separate working state from immutable snapshots
- support branching, diff, and merge
- record runs, packets, and metrics
- enable export, import, backup, and audit

## Overview

Target structure:

```text
.ctx/
  version.json
  config.json
  project.json
  HEAD
  branches/
  commits/
  graph/
  working/
  staging/
  runs/
  packets/
  index/
  metrics/
  providers/
  logs/
```

Not all folders contain files immediately. Some are populated only after specific flows run.

## Base files

### `version.json`

Describes repository format version.

Responsibilities:
- repository compatibility
- persisted format control

Example:

```json
{
  "currentVersion": "1.0",
  "initializedAtUtc": "2026-04-07T20:12:05.4352656+00:00"
}
```

### `config.json`

Repository configuration.

Responsibilities:
- default provider
- available providers
- packet token limit
- metrics tracking

Example:

```json
{
  "defaultProvider": "openai",
  "providers": [
    {
      "name": "openai",
      "defaultModel": "gpt-4.1",
      "endpoint": "https://api.openai.com/v1/responses",
      "enabled": true
    },
    {
      "name": "anthropic",
      "defaultModel": "claude-3-7-sonnet-latest",
      "endpoint": "https://api.anthropic.com/v1/messages",
      "enabled": true
    }
  ],
  "packetTokenLimit": 16000,
  "trackMetrics": true
}
```

### `project.json`

Represents the root `Project` entity.

Responsibilities:
- project identity
- default branch
- project state
- creation traceability

Example:

```json
{
  "name": "CTX",
  "description": "Cognitive Version Control System",
  "defaultBranch": "main",
  "state": 1,
  "id": {
    "value": "e92f9079e49a48a8a08bbeb324936ff6"
  },
  "trace": {
    "createdBy": "diegoxtr",
    "createdAtUtc": "2026-04-07T20:12:05.4303232+00:00",
    "updatedBy": null,
    "updatedAtUtc": null,
    "tags": [
      "project"
    ],
    "relatedIds": []
  }
}
```

### `HEAD`

Points to the current branch and last known commit.

Current format:

```text
main:676790c259864667940b503e6c7e5008
```

Responsibilities:
- currently checked-out branch
- fast reference to current commit

Rule:
- if there is no commit yet, commit may be logically `null`

## Directories

### `branches/`

Contains one reference per branch.

Example:

```json
{
  "name": "main",
  "commitId": {
    "value": "676790c259864667940b503e6c7e5008"
  },
  "updatedAtUtc": "2026-04-07T20:13:13.6697558+00:00"
}
```

Responsibilities:
- record branch pointers
- support `branch`, `checkout`, and `merge`

CLI relation:
- `ctx branch`
- `ctx checkout`
- `ctx merge`

### `commits/`

Contains immutable `ContextCommit` snapshots.

Responsibilities:
- reproducible history
- base for `log`, `diff`, `merge`, `export`

Snapshot rule:
- commits no longer snapshot only `WorkingContext`
- commits now persist a `RepositorySnapshot`
- `RepositorySnapshot` contains:
  - `WorkingContext` for active cognitive state
  - `Runbooks` for stable operational memory versioned with the repo
  - `Triggers` for stable origin memory versioned with the repo

Why this matters:
- `OperationalRunbook` stays outside mutable `working-context.json`
- `CognitiveTrigger` stays outside mutable `working-context.json`
- but still becomes part of CTX history, diff, merge, export, and import semantics

Invariant:
- a commit must not mutate after being persisted

CLI relation:
- `ctx commit`
- `ctx log`
- `ctx diff`

### `graph/`

Materialized view of the current cognitive graph.

Current file:
- `graph/current-graph.json`

Responsibilities:
- navigable snapshot of current domain state
- support inspection and future optimizations

Notes:
- coexists with `working/` as a useful operational state
- may evolve into a specialized projection

### `working/`

Contains the active `WorkingContext`.

Current file:
- `working/working-context.json`

Responsibilities:
- mutable in-progress state
- artifacts added but not necessarily committed
- base for `status`, `context`, `run`, and `commit`
- intentionally excludes stable operational memory such as `OperationalRunbook`

CLI relation:
- `ctx status`
- `ctx goal add`
- `ctx task add`
- `ctx hypo add`
- `ctx decision add`
- `ctx evidence add`
- `ctx conclusion add`
- `ctx run`
- `ctx context`

### `staging/`

Contains the snapshot ready to materialize into a commit.

Current file:
- `staging/staged-context.json`

Responsibilities:
- persistence of the staging snapshot
- reproducibility between working and commit

Notes:
- currently mirrors the snapshot persisted by the commit engine
- a base for finer staging in future versions

### `runs/`

Contains persisted AI runs.

Responsibilities:
- record `Run`
- preserve structured outputs per run
- enable operational inspection and analysis

CLI relation:
- `ctx run`
- `ctx run list`
- `ctx run show`

Notes:
- may be empty if no runs were executed

### `packets/`

Contains `ContextPacket` generated by `ContextBuilder`.

Responsibilities:
- persist optimized context sent to providers
- audit what information was prioritized in a run

CLI relation:
- `ctx context`
- `ctx packet list`
- `ctx packet show`

Notes:
- may be empty if no packets or runs were created

### `index/`

Reserved for search/access optimization.

Current state:
- contains a `README.txt` placeholder

Responsibilities:
- future indices
- faster local queries
- secondary repository projections

### `metrics/`

Contains operational and economic metrics.

Current file:
- `metrics/usage.json`

Operational note for this self-hosted repo:
- `metrics/usage.json` is treated as runtime telemetry and can be excluded from Git to avoid dirtying the working tree on read-only commands

Example:

```json
{
  "totalRuns": 0,
  "totalTokens": 0,
  "totalAcuCost": 0,
  "repeatedIterations": 0,
  "avoidedRedundancyCount": 0,
  "totalExecutionTime": "00:00:00"
}
```

Responsibilities:
- cumulative cost
- tokens
- avoided redundancy
- total execution time

CLI relation:
- `ctx metrics show`

### `providers/`

Reserved for provider-specific metadata, caches, or future projections.

Current state:
- contains a `README.txt` placeholder

Responsibilities:
- separate provider integration concerns
- enable extensions without polluting domain state

### `logs/`

Reserved for diagnostics and operational logs.

Current state:
- contains a `README.txt` placeholder

Responsibilities:
- troubleshooting support
- local operation logging
- possible technical audit

## Conceptual relationship between directories

Operational summary:

- `working/` = current mutable state
- `staging/` = snapshot ready or aligned for commit
- `commits/` = immutable history
- `branches/` + `HEAD` = navigation and current pointer
- `runs/` + `packets/` = AI interaction
- `metrics/` = cost/usage observability
- `graph/` = cognitive state projection

## Typical write flow

### 1. Initialization

`ctx init` creates:
- `version.json`
- `config.json`
- `project.json`
- `HEAD`
- folder structure
- `working/working-context.json`
- `graph/current-graph.json`
- `staging/staged-context.json`
- `metrics/usage.json`

### 2. Daily work

Commands like `goal add`, `task add`, `hypo add`, `decision add`, `evidence add`, `conclusion add`, and `run` mainly update:
- `working/`
- `graph/`
- `runs/`
- `packets/`
- `metrics/`

### 3. Cognitive commit

`ctx commit`:
- generates an immutable snapshot in `commits/`
- updates `HEAD`
- updates the active branch in `branches/`
- syncs `staging/`

### 4. Export and import

- `ctx export` serializes a portable repository state
- `ctx import` reconstructs `.ctx/` in another environment

## Recommended invariants

Keep these rules to preserve integrity:

- `HEAD` must point to an existing branch
- the current branch must exist in `branches/`
- `working-context.json` must always be deserializable
- commits in `commits/` must not mutate
- referenced IDs between entities must exist
- `metrics/usage.json` must remain cumulative and consistent
- if a repo versions `.ctx/` in Git, explicitly decide whether `metrics/usage.json` belongs to cognitive history or only local telemetry
- `config.json` must reflect valid providers recognized by the app

## What can be empty

It is normal for these folders to be empty at first:

- `runs/`
- `packets/`
- `index/`
- `providers/`
- `logs/`

That is not an error. It depends on executed flows and repo maturity.

## What should not be a primary source

By design, `.ctx/` should not use as primary source:

- raw chat logs
- full unstructured conversations
- embeddings as domain truth

The primary source is always the structured domain entities.
